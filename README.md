# test_task

В приложении реализована кастомная ролевая система управления доступом

Права доступа определяются на основе:
- роли пользователя
- бизнес-элемента системы
- набора разрешенных действий

Проверка доступа выполняется с помошью кастомного permission класса **ProjectPermission**

# Основные сущности системы

## Пользователь (User)
Модель пользователя **ProjectUser** содержит основную информацию о пользователе и его роль в системе.

### Основные поля
- email
- password
- first_name
- last_name
- role - ссылка на модель Role
- is_admin - флаг администратора системы
- is_active - флаг активности пользователя

# Роли пользователей
Модель **Role** определяет тип пользователя в системе.
### Примеры ролей
- admin
- manager
- user

Каждая роль определяет набор доступных действий через систему правил
доступа.

# Бизнес-элементы системы
Модель **BusinessElement** описывает объекты системы, к которым
применяется контроль доступа.

### Примеры элементов
- user - пользователи системы
- roles - роли пользователей
- permissions - правила доступа
- elements - элементы системы

Каждый элемент используется при проверке доступа в **ProjectPermission**.

Во Вью-функциях указывается элемент, к которому применяется проверка:
``` python
element = 'user'
```

# Правила доступа
Модель **PermissionRoleRule** определяет, какие действия может выполнять
роль над конкретным элементом системы.
Каждая запись определяет **права одной роли для одного элемента системы**.

### Ограничение уникальности
Для каждой пары: (role, element) может существовать только **одно правило доступа**.

# Типы разрешений
В модели **PermissionRoleRule** реализованы следующие разрешения.
## Чтение
- read_permission - разрешает читать **свои объекты**
- read_all_permission - разрешает читать **любые объекты**

## Создание
- create_permission - разрешает создавать объекты.

## Обновление
- update_permission - разрешает редактировать **свои объекты**
- update_all_permission - разрешает редактировать **любые объекты**

## Удаление
delete_permission - разрешает удалять **свои объекты**
delete_all_permission` - разрешает удалять **любые объекты**


# Проверка доступа

Проверка прав реализована в классе: **ProjectPermission**

Проверка выполняется в два этапа.
# 1.Проверка доступа к ресурсу

Метод **has_permission**

Проверяет:
- Пользователь аутентифицирован
- Определяется элемент системы (view.element)
- Получается правило доступа:
- Проверяется наличие разрешения для соответствующего HTTP-метода.

### Соответствие методов и разрешений

  GET          read_permission / read_all_permission
  POST         create_permission
  PUT          update_permission / update_all_permission
  PATCH        update_permission / update_all_permission
  DELETE       delete_permission / delete_all_permission

Если пользователь имеет флаг **is_admin**, доступ разрешается независимо
от правил.

# 2.Проверка доступа к объекту

Метод **has_object_permission**

Проверяет:
- является ли пользователь администратором
- есть ли разрешение на работу **со всеми объектами** или пользователь работает **со своим объектом**


# Аутентификация

В системе используется JWT-аутентификация реализованна в классе **JWTAUTH**.

### Алгоритм работы
1.  Токен извлекается из заголовка запроса:
2.  JWT токен декодируется
3.  Из токена извлекается user_id
4.  Выполняются проверки:
- пользователь существует
- пользователь активен
- токен отсутствует в таблице**DeactivatedToken**

Если проверка не проходит, возвращается ошибка: 401 Unauthorized

# Выход из системы

При выходе пользователя токен добавляется в таблицу **DeactivatedToken**

Это предотвращает повторное использование токена после выхода из
системы.

# Коды ошибок
Система возвращает стандартные HTTP-коды ошибок:
- 401 Unauthorized - пользователь не аутентифицирован.
- 403 Forbidden пользователь аутентифицирован, но не имеет прав доступа к запрашиваемому
ресурсу.

# Административный API
Администратор может управлять системой прав доступа через API.
Доступны следующие операции:
- CRUD для Role
- CRUD для BusinessElement
- CRUD для PermissionRoleRule

Это позволяет **динамически управлять системой прав без изменения кода приложения**.

## Установка и запуск
### 1. Клонирование репозитория
```bash
git clone https://github.com/BocharnikovSergey/test_task
cd test_task
```

### 2. Создание виртуального окружения
Cоздать и активировать виртуальное окружение:

* Если у вас Linux/macOS
```bash
    python3 -m venv env
    source env/bin/activate
    python3 -m pip install --upgrade pip
```
* Если у вас windows
```bash
    python -m venv env
    source env/scripts/activat
    python -m pip install --upgrade pip
```

### 3. Установить зависимости из файла requirements.txt:
```bash
pip install -r requirements.txt
```

### 4. Выполнить миграции:
* Если у вас Linux/macOS
```bash
    python3 manage.py migrate
```
* Если у вас windows
```bash
    python manage.py migrate
```

### 5. Загрузка данных
```bash
python manage.py seed_data
```

### 6. Запуск сервера
```bash
python manage.py runserver
```
